## Objetivo De Esta Version
Implementar un flujo formal para agregar adherentes a ventas ya completadas mediante anexos posteriores a la firma del contrato principal.

Esta version se diseno para:
- permitir altas posteriores en ventas `empresarial` y `unipersonal`
- respetar la vigencia del contrato padre
- obligar auditoria antes de activar adherentes
- separar el flujo de firma del anexo del flujo de firma principal del contrato
- dejar trazabilidad suficiente para que futuras migraciones a `main` y Brasil sean controladas

## Resumen Funcional
El sistema ahora permite crear un anexo sobre una venta ya completada. Ese anexo contiene uno o mas adherentes nuevos. El anexo pasa por auditoria, luego genera documento y link de firma, y solo cuando la firma del anexo queda completada el adherente pasa a estado activo dentro de la venta padre.

Los adherentes dados de alta por anexo:
- quedan vinculados al anexo con `source_addendum_id`
- usan la misma fecha de vencimiento del contrato padre
- no deben editarse ni eliminarse directamente desde los componentes normales de adherentes

## Flujo De Negocio
1. La venta padre debe estar en estado `completado`.
2. La venta debe tener `sale_type` igual a `empresarial` o `unipersonal`.
3. La venta no debe estar vencida. Se usa `contract_end_date`.
4. Desde `SaleDetail` se crea un anexo nuevo.
5. El anexo empieza en `borrador`.
6. Se cargan adherentes en `sale_addendum_beneficiaries`.
7. El anexo se envia a auditoria mediante RPC.
8. Un auditor, admin, supervisor o super_admin aprueba o rechaza.
9. Si se aprueba:
- se crean beneficiarios en `beneficiaries`
- si requieren firma, quedan en `pending_addendum_signature`
- se crea un documento de tipo `anexo_alta_adherente`
- se crean `signature_links` asociados al anexo
- el anexo pasa a `enviado_firma`
10. El adherente firma desde el link publico.
11. El backend completa el anexo, activa el beneficiario y recalcula el total de la venta.
12. Cuando no quedan firmas pendientes, el anexo pasa a `completado`.

## Reglas Tecnicas Importantes
### Vigencia contractual
- al completar una venta, se calcula `contract_completed_at`
- `contract_end_date` se define a 1 ano desde la fecha de completado
- un anexo no puede aprobarse si la venta esta vencida

### Separacion de flujos
El flujo de firma de anexos no debe activar pasos del contrato principal.

Regla tecnica:
- `signature_links` de anexo usan `sale_addendum_id`
- `finalize-signature-link` filtra el flujo principal con `sale_addendum_id IS NULL`
- los links de anexo no deben reactivar la firma de `contratada`

### Documento de anexo
El documento generado para el alta posterior usa:
- `document_type = 'anexo_alta_adherente'`

Esto evita mezclar anexos con `ddjj_salud` del flujo original.

### Activacion del adherente
Un adherente del anexo se activa solo si:
- el link pertenece al anexo correcto
- el link esta `completado`
- el link esta activo
- el link no esta revocado
- el link no esta expirado
- el anexo esta en `enviado_firma`

## Problemas Detectados Y Corregidos
### Corregidos
- el frontend podia enviar a auditoria con update directo
- el flujo de firma de anexos podia mezclarse con el flujo principal de firma
- el documento del anexo se registraba como `ddjj_salud`
- los adherentes activados por anexo podian editarse o borrarse desde UI normal
- faltaban estados de error y retry en el panel de anexos
- el link publico podia reportar exito aunque el anexo no quedara completado correctamente
- RLS era demasiado permisivo para escritura

### Riesgos que esta version intenta contener
- anexos cruzados entre companias
- re-aprobaciones fuera de estado
- activacion por links revocados o vencidos
- inconsistencias por beneficiarios con `status` nulo
